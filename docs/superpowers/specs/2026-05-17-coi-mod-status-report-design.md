# Captain of Industry Mod Status Report 设计文档

- 日期：2026-05-17
- 阶段：设计确认版（Web MVP 优先）
- 目标：先输出可部署公网的 Web 版，再为未来本地增强版预留演进接口

## 1. 项目概述
Captain of Industry Mod Status Report 是一个面向 Captain of Industry 玩家的网站工具。第一阶段目标不是做本地启动器或桌面管家，而是做一个公网可访问、零安装、低门槛的 Mod 状态查询站点。用户通过浏览器手动选择本地 Mods 目录后，系统读取各 Mod 的 `manifest.json`，提取本地版本信息，并与 COI Hub 上的远程版本结果进行对比，帮助用户识别可更新项、匹配异常项和待确认项。

该产品第一阶段明确坚持 Web 边界：不自动读取用户默认游戏路径，不直接替换本地文件，不做下载覆盖、不做冲突修复、不做启动器级管理。与此同时，架构层要为未来的 Tauri / 本地 helper 增强版预留稳定扩展口，避免后续推倒重来。

## 2. 产品目标
1. 为玩家提供一个公网可访问、零安装的 Mod 状态查询工具。
2. 帮助用户快速对比本地 Mod 信息与 COI Hub 远程版本状态。
3. 在不修改本地文件的前提下，建立未来向“本地增强版 / 下载替换能力”演进的清晰边界。

## 3. 目标用户
- 有多个 Mod，希望定期确认版本状态的普通玩家
- 需要排查 Mod 是否过期、来源是否异常的活跃玩家
- 未来可能愿意安装本地增强版进行一键升级的进阶用户

## 4. 范围定义
### 4.1 P0 必须能力
- 选择整个 Mods 目录作为主入口
- 扫描目录并读取每个 Mod 的 `manifest.json`
- 解析并展示 `id`、`version`、`display_name`、`authors`
- 基于 `display_name` 查询 COI Hub 搜索结果
- 优先进行精确匹配
- 精确匹配失败时展示候选项供用户手动确认
- 展示本地版本、远程版本、匹配状态、更新状态
- 支持手动触发“检查全部 Mod 版本”

### 4.2 P1 应该能力
- 提供结果筛选：全部 / 有更新 / 未匹配 / 已手动确认
- 保留当前会话中的人工匹配结果，避免重复选择
- 显示检查时间、数据来源和结果误差提示
- 支持撤销已手动确认的绑定关系

### 4.3 P2 可选能力
- 导出检查结果摘要
- 增加未来“下载 / 替换”入口占位
- 提供面向本地增强版的升级引导说明

### 4.4 非目标
- 不自动读取 `%USERPROFILE%\AppData\Roaming\Captain of Industry\Mods`
- 不直接下载、覆盖、删除或替换本地 Mod 文件
- 不处理复杂依赖修复、冲突诊断或启动器级管理
- 不承诺 COI Hub 搜索结果绝对唯一、绝对准确

## 5. 推荐架构
推荐采用 **Web MVP 前端主导 + 明确边界适配层** 的三层结构：

1. **核心领域层**
   - Mod 解析与标准化
   - Hub 匹配决策
   - 版本比较
   - 更新状态计算
   - 统一状态 / 错误枚举

2. **基础设施适配层**
   - 浏览器目录访问适配
   - COI Hub 查询适配
   - 会话缓存与绑定持久化
   - 未来 Tauri / helper 通信适配

3. **界面编排层**
   - 导入与权限引导
   - 列表展示
   - 候选绑定
   - 筛选与重查
   - 风险提示与状态反馈

该架构的关键价值是：Phase 1 只做纯 Web 可部署能力；Phase 2 若接入 Tauri 或本地 helper，只替换目录访问与本地执行适配器，而核心规则与大部分 UI 可以保留。

## 6. 前端模块设计
### 6.1 Onboarding / Import 模块
负责展示产品能力边界、浏览器限制说明和“选择整个 Mods 目录”入口。未授权前不展示伪结果。

### 6.2 Scan / Manifest 模块
负责递归查找各子目录中的 `manifest.json`，解析后生成标准化本地 Mod 列表。若某个目录结构异常或 manifest 损坏，只标记该条失败，不阻断整体扫描。

### 6.3 Hub Search 模块
负责基于 `display_name` 查询 COI Hub，输出统一远程候选结果模型，并根据匹配规则给出精确匹配或候选待确认状态。

### 6.4 Binding 模块
负责处理自动匹配、人工候选选择、撤销重选与会话内绑定结果保留。手动确认后的绑定需在界面上明确可见。

### 6.5 Status Report 模块
以列表形式展示每个 Mod 的本地信息、远程版本、匹配状态、更新状态、最后检查时间和错误原因。

### 6.6 Filter / Action 模块
支持结果筛选，以及“检查全部 Mod 版本”的显式操作。该按钮只触发一次前台重查，不暗示后台持续监控。

### 6.7 Adapter 模块
统一封装 Web 文件访问、Hub 查询与未来本地增强能力，避免业务逻辑直接依赖浏览器或 Tauri 细节。

## 7. 核心数据模型
### 7.1 LocalMod
- `id`
- `version`
- `displayName`
- `authors`
- `manifestPath`

### 7.2 RemoteModCandidate
- `hubId`
- `title`
- `remoteVersion`
- `url`
- `matchType`

### 7.3 ModBinding
- 本地 Mod 标识
- Hub 条目标识
- `source = auto | manual`

### 7.4 ModStatusRecord
聚合本地信息、远程信息、匹配状态、更新状态、最后检查时间、错误信息，是最终驱动列表渲染的核心记录模型。

### 7.5 SessionState
保存当前目录上下文、人工绑定结果、筛选条件和临时缓存。

## 8. 匹配与状态规则
### 8.1 匹配规则
1. 使用 `display_name` 查询 COI Hub
2. 若标题精确一致，则自动匹配
3. 若无精确一致结果，则进入候选待确认状态
4. 用户手动确认后，记录为手动绑定
5. 用户可撤销绑定并重新选择

### 8.2 匹配状态枚举
- 已精确匹配
- 候选待确认
- 已手动确认
- 未匹配
- 查询失败

### 8.3 更新状态枚举
- 已是最新
- 疑似有更新
- 无法判断

其中“无法判断”适用于未匹配、远程异常或版本号无法可靠比较的情况，避免误报升级。

## 9. 核心数据流
1. 用户打开页面，看到能力边界说明
2. 用户手动选择整个 Mods 目录
3. 系统扫描并解析本地各 Mod 的 `manifest.json`
4. 形成标准化 `LocalMod` 列表
5. 按 `display_name` 查询 COI Hub
6. 精确匹配则自动绑定，否则生成候选列表
7. 用户对待确认项进行人工选择
8. 状态计算器输出最终 `ModStatusRecord`
9. 列表模块展示结果并支持筛选
10. 用户点击“检查全部”时，仅刷新远程查询与状态结果，不重新要求导入目录

## 10. 异常与边界处理
- **浏览器不支持目录访问 API**：提示当前环境不支持 Web MVP 核心能力
- **目录选错或权限撤销**：提示重新选择整个 Mods 目录
- **manifest 缺失 / 损坏 / 字段不全**：单条标记异常，不阻断其他 Mod
- **Hub 搜索失败 / 页面结构变化 / 限流**：标记外部依赖异常，支持稍后重试
- **display_name 重名、特殊字符、多语言差异**：禁止自动绑定，进入人工确认
- **版本号不规范**：回退为“无法判断”，不强行得出可更新结论

## 11. 演进路线
### Phase 1：Web MVP
完成目录导入、manifest 解析、Hub 精确匹配、候选人工确认、状态展示、检查全部、基础筛选、会话内绑定保留和风险提示。

### Phase 2：本地增强版
在不重写核心领域层的前提下，引入 Tauri 或 helper，补充：
- 默认路径快捷导入
- 更稳定的目录扫描
- 下载 / 替换入口
- 迁移引导与更强本地能力

## 12. 可复用核心层清单
以下部分必须从一开始就抽象为可复用核心层：
1. Manifest 解析与本地 Mod 标准化
2. Hub 搜索结果到匹配决策的规则
3. 版本比较与更新状态判定
4. 绑定关系模型与持久化接口抽象
5. 错误枚举、状态枚举和统一结果模型

## 13. 结论
本项目应坚持“**Web 先行、适配隔离、领域内聚、升级有路**”的设计原则。第一阶段只建立状态识别、版本对比和人工确认流程，不越界操作本地文件；未来再通过 Tauri 或本地 helper 平滑扩展为真正具备下载与替换能力的增强版工具。
